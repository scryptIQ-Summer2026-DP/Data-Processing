    Learning Objectives

    Introduction

    In this section we will work through the ITSN1_N expression data that we discovered was rightwards skewed after our attempts to remove outliers using Z-score analysis. Here we will address how to transform this type of data into a more normal distribution, both through an interpretable method — log transformation — and a distribution-agnostic method — Yeo-Johnson transformation.

    Setup

    Normalising - log

    If your data looks like and is known to increase in its units as a fold-change (multiplicative) rather than absolute amounts (additive) - it is more natural to work on a log scale. Working with log means the unit axis become fold-changes rather than absolute, the distribution becomes more symmetrical (approximating normality better), and the variance becomes more stable across the range of values. This matters because some tools either assume normality or behave better under it.

    For example, take Z-score from the first section. When interpreting the scores we would say anything $\pm3$ units would be an outlier as that contains 99.7% of the values. This assumption does not hold true when applied to right-skewed data, as a higher percentage would be outside this boundary.

    Examples of when a tool assumes the data is normally distributed:

    • Confidence intervals around the mean

    • Parametric P-value tests: T-test, ANOVA

    • Gaussian Naive Bayes (supervised ML)

    • Gaussian Mixture Models (unsupervised ML)

    Examples of when a tool performs better when the data is normally distributed:

    • Principal Component Analysis (dimensionality reduction)

    • k-nearest neighbours (supervised ML)

    • support vector machines (supervised ML)

    • neural networks (supervised ML)

    We will be covering the machine learning models in the Classical ML and AI module.

    KEY TERMS

    Logarithmic: Describes a relationship where values change by multiplication rather than addition. On a logarithmic scale, each equal step represents a constant fold-change (e.g. ×10) rather than a constant amount. A quantity is said to be "logarithmic" or "log-normal" when it varies naturally as ratios.

    Log-scale: An axis or transformation where the data are represented by their logarithms, so that equal distances correspond to equal ratios rather than equal differences.

    Transforming with log

    To transform a log-normal set of data, you need to apply a logarithm to all elements. The standard one to use is natural log or log in base e ($log_e$). Alternatively, you can use $log_2$ or $log_{10}$. Each one describes the fold change in terms of their base, i.e. with $log_2$ & $log_{10}$ each unit step describes a 2- or 10-fold change respectively. For $log_e$, each step is an e-fold change, i.e. ~2.718 times.

    Additionally, the values at the end of the tail to the right have also been drawn closer to the bulk of the distribution.

    The width of each distribution is an inverse of the size of the fold change it represents, with base 10 having the smallest range as many more values fall within each step increase, as greater increases are needed to have a fold-change of 10.

    However, plotting them on their own scales (below) it can be seen that the shapes are still identical; it is just their scale (x-axis) that differs.

    One caveat to log transformations is that 0 can not be transformed and will instead return negative infinity and a Runtime Warning with the NumPy function.

    Zeroes are common in biological datasets, especially when cleaned for background noise. If you have zeroes, to counter the data being lost you can add a pseudocount to the data to bring them above 0. The standard addition is 1. NumPy has a function np.log1p() that automatically adds 1 before applying a natural log. However, if you want to use $log_2$ or $log_{10}$ you can simply add 1 to the data yourself.

    Adding 1 actually bunches up much more of the lower values, as an increase of 1 for smaller values is much greater than those above them, recapitulating slightly the steep increase just after zero seen in the untransformed data. This is because adding 1 puts everything into the 1-2 range where log is nearly linear, thus not transforming it much at all.

    One thing to note: when transforming data due to the necessity of the downstream tool - such as machine learning tools - the base is irrelevant. All that matters is that the distribution is more normal to fit the tools assumptions. The base matters when you want the data to be more interpretable for humans, such as $log_2$ for expression data, as doubling every step is inherently easy for a human to understand visually.

    Normalising - Yeo

    Log is useful when we know the underlying mathematics of the data, i.e. it should follow a logarithmic scale. Yet, this information is not always known and we still need to transform the data to be approximately normal, such as in supervised machine learning.

    One tool (of many) is the Yeo-Johnson transformation, which - like log transformation - stabilises the variance across the range of values and pulls the distribution towards symmetry. However, unlike the log transform, Yeo-Johnson makes no strong assumption about the underlying scale of the data. In fact, the tool will find the value $\lambda$ that best transforms the data to normal through a process called maximum likelihood which we will explore in the Classical ML and AI module. For now, it is a method that estimates the best parameter (in this case $\lambda$) that will give you your desired result (in this case a normalised set of data) through an iterative process.

    FACT

    Yeo-Johnson is a generalisation of the Box-Cox transformation. The difference being that it can handle positive and negative values, unlike the latter.

    We will be implementing the transformation in two ways:

    1. As standard using scipy's Yeo-Johnson function

    2. With scikit-learn's PowerTransformer function which also scales the data to have a mean of zero and standard deviation of 1 (unit normal distribution)

    NOTE

    scikit-learn is an all-encompassing machine learning package in Python. We will be exploring it in much greater detail in the Classical ML and AI module.

    Both return effectively the same $\lambda$, confirming they implement the same underlying method — the only difference is that PowerTransformer also standardises the output to a standard normal distribution. Plotting them side-by-side you can see the implementation without scaling is tightly clustered just above zero in contrast to the scaled data which has the majority of its data (68%) between -1 and 1.

    Placing them onto separate axes to prevent the visual squashing you can now see that their shape is exactly the same - just as we saw with the log values - it is just the range they occupy that is different. This is a key observation: the data shape is not changed by the scaling, it is just shifted and rescaled to fit a standard range.

    Standardising puts every feature onto a common scale (in standard-deviation units), so that no single feature dominates a model simply because its raw values happen to be larger. This makes features of varying scales and shapes much more comparable and equitable inside the model's training.

    The decision of when to log transform versus using a tool agnostic of the underlying principles lies in your use case. If you want to retain interpretability, use log, as the values remain meaningful as fold-changes. Something like Yeo-Johnson, by contrast, trades all interpretability for automatic normalisation, to help downstream tools perform correctly or better.

    Summary & Quiz

    This section tackled skewed features and why normality matters for so many statistical and machine-learning tools. The log transform suits strictly positive, multiplicative data and keeps values interpretable as fold-changes, while the Yeo-Johnson transform normalises without assuming an underlying scale and copes with zero and negative values. We also saw how standardisation rescales a feature to zero mean and unit standard deviation, and revisited our Z-score outliers now that the skew had been addressed.
